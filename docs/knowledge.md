実際の使用で得た知見をここに蓄積する。
テンプレートや CLAUDE.md への反映候補もここに記録する。

## AI ごとの特性（実地知見）

### Web Gemini（PM役・設計役）

**良かった点:**

- テンプレートのおかげで「整理モード」が機能した
- 推測で突き進まず、質問してくれるようになった
- Gem 化すると毎回コピペ不要で便利

**課題:**

- 添付資料（Google ドキュメント・NotebookLM・GitHub リポジトリ）を読んでくれないことがある
  → 対策：NotebookLM で事前にリサーチして、その結果テキストを直接貼り付ける
- 長い会話でパンクする
  → 対策：PM役と設計役を必ず別チャットに分ける
- **裏取りしない**：最初の答えで満足してしまい、検証や深掘りが浅い
- **YES マン傾向**：ユーザーを怒らせないように必死にヨイショする。結果、発展性のない回答になりがち（言われたことしかやらない＝マイクロマネジメント状態）
- **Gemini3 以降で劣化感**：知識はアップしているが、批判的思考・反対意見が出にくくなった印象

**YES マン問題の根本仮説**:

モデル本体の問題というより、**モデル前段の調教（システムプロンプト・RLHF の副作用）**に起因する可能性が高い。
以下の根拠:

- モデル自体にはパワーがある（強烈なプロンプトで上書きできればちゃんと動く）
- テンプレート（`gemini_pm.md` / `gemini_design.md`）の「禁止事項」節（推測での断定禁止・承認なしに進まないなど）を効かせると挙動が大きく改善する
- つまり**テンプレが調教を上書きしにいく**ことで本来のパワーが引き出せる

**対策**: 強めの禁止事項プロンプトでシステム側の「ヨイショ傾向」を上書きする。既存の Gemini 役テンプレはこの方向で設計されている。

**解釈（実運用からの観察）**:

Gemini の弱点は「文章読解力」ではなく「曖昧プロンプトへの補完能力」にある可能性が高い。
整理すると各 AI の役割がきれいに分かれる:

- **Claude Code**: 曖昧 → 具体化に強い（リポジトリ文脈を加味した補完）
- **Gemini**: 具体 → 整理・大量処理に強い（長文読解・ドキュメント統合）
- 直列でつなぐと、人間の曖昧プロンプトが高品質な仕様書に変換される

AI チーム化の本質的価値は「各 AI の強みを活かす」だけでなく「**各 AI の弱みを別の AI で補う**」点にあるとも言える。

### Claude Code（実装役）

**良かった点:**

- 「任せて安心」という感覚
- 指示の意図をよく理解する
- GitHub Issues との連携が強い

**課題:**

- レート上限（5時間枠）にすぐ達する。2セッションで計6〜7時間弱
- コンテキストが溜まると上限到達が早くなる
- 初期プランが詳細で良いのにファイルに残らない（AAR より精緻）

**対策:**

- 初期プランを必ず `handoff/plan_YYYYMMDD.md` に保存させる
- タスクを小さく分割してセッション単位のコンテキスト量を抑える

### GitHub Copilot（補完役）

**良かった点:**

- Claude Code のテンプレートを使い回せた
- 細かい補完に強い

**課題:**

- 要件理解が弱い。Claude Code の初期プランがあると大幅に改善

**対策:**

- `handoff/plan_YYYYMMDD.md` を Copilot に見せてから作業させる
- Claude Code レート上限時のフォールバックとして活用

## ワークフロー上の発見

### リサーチャー役の有効性

NotebookLM + DeepResearch で事前にソースを追加し仕様を洗い出した後、
Gemini PM役に渡すフローが有効だった。

```text
NotebookLM（リサーチ） → 仕様メモ → Gemini（PM役） → spec.md
```

つまずきポイントや実現可能性を事前にインプットできる。
→ リサーチャー役テンプレートとして整備する価値あり

### リサーチは「オプション」ではなく最重要フェーズ（会社運用でわかったこと）

会社リポジトリで運用してみて判明: リサーチは手戻りしてでもやる価値がある。
ハマりどころはリサーチで見つかるため、コーディング中に詰まった場合でもリサーチに戻るのが正解。

**精度が上がった具体パターン:**

1. **Claude Code で下書きをブラッシュアップ**:
   - 自分でリサーチプロンプトの下書きを書く（粗くてよい）
   - リポジトリ文脈を持つ Claude Code にブラッシュアップさせる
   - 詳細条件がびっちり埋まり、DeepResearch の精度が大きく上がる
   - 推奨設定: Opus + Effect MAX + Thinking オン + Superpowers オン

2. **コーディング中に詰まったら「調査プロンプトの生成」を依頼**:
   - Claude Code に直接調査させるとトークン消費が激しい
   - 代わりに「Gemini DeepResearch 用のプロンプト作って」と依頼する方がトークン効率が良い
   - 生成プロンプトを DeepResearch に投げ、結果を Claude Code に戻す

3. **Gemini にはファイル内容を直接混ぜる**:
   - Gemini はリポジトリ・コード添付を読まないことがある
   - Claude Code が生成するプロンプトでは「ここにこう書いてある」と具体的なファイル内容を混ぜ込む

4. **一次情報で裏を取る**:
   - DeepResearch や NotebookLM は偏った情報・古いブログ記事の孫引きに引きずられる
   - GitHub リポジトリのアーカイブ状態・最終コミット日・Issue の活発度を必ず確認する
   - 実例: Prisma の Python クライアント（prisma-client-py）は検索結果では推奨されるが、GitHub リポジトリではアーカイブ済み（開発終了）。DeepResearch だけでは気づけない
   - Claude Code に「DeepResearch の結果を GitHub Issue でも裏付けとって」と依頼するのが有効

### handoff/ はコミットすべき

gitignore すると知見が失われる。

- 作業ログとして価値がある
- チームで共有すると有用
- AAR から改善事項を吸い上げ、テンプレートに反映するフィードバックループに使える

会社では `handoff/` をリポジトリに含めて運用。うまく機能した。

### 初期プランを必ず保存する

Claude Code がセッション開始時に出す「状況把握レポート＋作業計画」が最も詳細。
AAR より精緻なのでここを失うのはもったいない。

**対応:** `handoff/plan_YYYYMMDD.md` として保存を CLAUDE.md に明記する。

### CLAUDE.md の命名問題

チーム開発リポジトリでは `CLAUDE.md` をルートに置きにくい。

- 会社では `NOTE.md` という名前にして運用（Claude Code は任意ファイルを読める）
- セッション開始時に `「NOTE.md を読んでルールを把握して」` と指示

### ブランチ・コミット規約

AI がブランチを切らずに main 直接作業するリスクがある。
プロジェクトごとにコミット規約・ブランチ規約が異なる。

**対応:** CLAUDE.md にブランチ作成を必須化し、規約不明時は確認させる。

### モデルディスパッチャパターン（試行中）

会社の Claude Code 運用で回している運用パターン。
フェーズ 2 の「ウォーターフォール問題への対応」の実装例にもなる。

**パターン**:

1. **Opus** にタスクを相談
2. Opus がプラン提示 → 「これなら Haiku / Sonnet で回せます」と振り分けを提案
3. **Sonnet / Haiku** で実装
4. 詰まったら Opus が **Gemini DeepResearch 用プロンプト** を生成
5. DR 結果を Sonnet Thinking オフ で処理
6. 詰まったら Opus に戻す

**役割分担の整理**:

- **Opus** = メタオーケストレーター（判断・ディスパッチ・復旧）
- **Sonnet / Haiku** = ワーカー（軽量実装）
- **Gemini DeepResearch** = 外部調査員（最新情報・広範リサーチ）

**既存テンプレとの関係**:

- `ai-team/templates/researcher.md` の「コーディング中に詰まったとき」節は既にこのパターンの一部をテンプレ化している（Gemini DR プロンプト生成）
- サブエージェント（ `.claude/agents/` ）で外部 CLI をラップする案もあるが、**エスカレーション時のコンテキスト引き継ぎロスが大きい**ため、対話型（Opus が毎回ディスパッチ判定）の方が適している

**現状のステータス**:

**ログ蓄積フェーズ**。実運用でこのパターンが何度回ったか、どんな判断が発生したかを記録中。
パターンが安定したら以下の順で昇格を検討:

1. 本節にログを追記していく（現在ここ）
2. `ai-team/NOTE.md` に「モデル選択」の節を数行で追加
3. `ai-team/templates/dispatcher.md` を新設（時期尚早なら見送り）

#### ログ: 2026-04-25 反例（規約該当判定タスクで Sonnet 破綻）

**文脈**: 顧客の社内規約をベースに、受託システムのセキュリティポリシー該当箇所を一次仕訳するタスク。NotebookLM で抽出した md の精査フェーズ。

**起きたこと**:

- 昨日（2026-04-24）は Opus で初期プラン＋最初の抽出まで順調
- 今日、後続処理を Sonnet に振った瞬間に自動コンパクト発動（会社設定の Sonnet は 200K のみ）
- 文脈喪失で精査結果がめちゃくちゃに
- さらに「該当判定」「主語のゆらぎ」系タスクは Sonnet では精度が出ないことが判明

**この反例から見えた「モデル選びの3軸」**:

| 軸 | Sonnet/Haiku OK | Opus 必須 |
|---|---|---|
| 難易度 | ルーチン処理 | 複雑な推論 |
| コンテキストサイズ | 200K 以内で完結 | 大量文書の一貫処理 |
| 判断の抽象度 | 機械的変換（整形・整理） | 該当判定・分類・矛盾検出（主語がゆらぐ） |

**ワーカー選定の判断規則**:

- 境界テスト：人間が読んで「ん？これどっちだろう」と迷う種類のタスクは Opus 案件
- コンテキスト超過の兆候（自動コンパクト発生）が出たら即 Opus へフォールバック
- 判断系 × 大量文書の組み合わせは **Opus 全振り前提で設計**

#### ログ: 2026-04-25 セッション分離の一般化（同一モデル内でも必要）

**気づき**: AI チームワークフロー＝「Gemini と Claude Code を分ける」話だと思っていたが、**本質は「コンテキストウィンドウを汚さないためにセッションを分ける」**こと。

相手が別 AI でも、同じ Opus でも、コンテキスト汚染の問題は同じ。

| レベル | セッション分離の意味 |
|-------|---------------------|
| AI 間（Gemini PM → Claude Code 実装） | 役割分担＋コンテキスト分離 |
| モデル間（Opus プラン → Sonnet ワーカー） | コスト最適化＋コンテキスト分離 |
| **同一モデル内（Opus プラン → Opus ワーカー）** | **コンテキスト分離のみ** |

**実運用ルール**:

- Opus 全振りでも **handoff/ 経由でセッションを分ける**のが基本
- 迷走したら手元で巻き戻すのではなく、**新セッションで再開**
- 同一セッションで外部ログ（Google ドキュメント等）を継ぎ足す運用は、履歴全体が毎ターン再送されるためトークンを無駄に消費する（＝今日の失敗）

##### 補足1: 「同一セッション内でのモデル切替」が招く自動コンパクト

今日（2026-04-25）の失敗の実際のメカニズムはこれ。同一セッション（チャットスレッド）内で発生した:

1. Opus でプランと序盤を進めた（コンテキストウィンドウ 1M）
2. 後続処理を Sonnet に切り替えた瞬間、**Sonnet のコンテキストウィンドウ（200K）を超過**
3. Claude Code が **自動コンパクト**（圧縮）を実行
4. 昨日の詳細・前段のやり取りがコンテキストから失われる
5. AI は「何をしたかったのかわからない」状態に（疑いもせず、質問も確認もせず、想像で迷走）

**教訓**: **モデル切替は新セッションと同等のコンテキストリセット**として扱うべき。
小さいウィンドウのモデルに切り替えるときは、**その時点で handoff/ ファイル化しておかないと、切替直後に文脈が消える**。

##### 補足2: 「同じセッションで外部ログ追加」と「別セッションで同じログを渡す」の違い（一般論）

自動コンパクトが走らなくても、同一セッションで履歴を継ぎ足していくと仕組み上の差がある:

| 観点 | 同じセッションで追加 | 別セッションで渡す |
|------|------------------|-----------------|
| 再送されるもの | 過去の迷走履歴 **＋** 新しいログ | 必要なログ **のみ** |
| コンテキスト汚染 | あり（迷走の残滓が残る） | なし（クリーン） |
| 判断品質 | 汚染された文脈で劣化 | クリーンな文脈で回復 |

外部ログ自体のトークン量は同じでも、**迷走履歴ぶんの無駄な再送がなくなる**のと、**汚染されていない文脈で判断できる**のが別セッションのメリット。

#### ログ: 2026-04-25 今日の失敗の完全な経緯（詳細ナラティブ）

同じパターンを回避するため、今日の連鎖失敗を時系列で記録する。

1. 同一セッション内で Opus → Sonnet に切り替え（後続処理を Sonnet に振るため）
2. Sonnet のコンテキストウィンドウ（200K）を超過、**自動コンパクトが発動**
3. Sonnet が迷走（昨日の前提・趣旨が圧縮で失われている）
4. Opus に戻すも精彩に欠ける（既に圧縮された文脈から回復できない）
5. プランを確認しようとしたら **「全体像・判断軸」が MD に書かれていない**（フェーズ1の詳細のみで、趣旨・概要・フェーズ2 の輪郭が欠落）
6. Google ドキュメントにログ退避していたので、MD 化して読ませる
7. NOTE.md の記載に従って Gemini CLI で要約を読もうとする
8. 感心して見ていたら、プロンプトが超適当でアウトプットがデタラメ
9. 自分で読めと指示して復帰
10. プランが曖昧だったことが判明
11. リプランニング
12. フェーズ 1 の精査
13. フェーズ 2 の抽出
14. フェーズ 2 の精査でまた Sonnet に振る → 迷走（暴走）
15. またログを読ませる（ただし **ログの先頭部分だけ切り出して渡す**。先頭には「経緯・やりたいこと・最初のプランニングログ」があるため、文脈復旧に最も効率的）

**この連鎖から見える教訓**:

- **モデル切替（特に小さいウィンドウへ）は自動コンパクトを発動させるリセット操作**として扱う。切替前に handoff/ ファイル化が必須
- 初期プランには **「全体像（趣旨・概要・全フェーズの輪郭）」＋「判断軸（なぜそう選んだか・なぜそうしなかったか）」** を含める。フェーズ1詳細だけでは不十分
- `gemini -p` 系の Gemini CLI 呼び出しも、**超適当なプロンプトを使わせると出力が汚染される**。中間成果物として読ませるなら、プロンプトの質を自分で確認する
- 迷走したら「過去履歴込みで巻き返す」より、**必要な部分だけ切り出して新セッション**のほうが早い（セッション分離の一般化原則）

##### 補足: プランニングの「思想」と「手順」の使い分け（観察）

このケースには **テンプレ設計の反省点** も含まれている。

- プランニングの草案（会話上）には全体像があった。でも **ファイル化されなかった**
- フェーズ1の詳細は書かれた（Sonnet/Haiku が迷わないように）
- しかし「なぜそう選んだか」の判断軸は MD のどこにも残らなかった
- 結果、**精緻な手順書になったが、思想の記録にはならなかった**
- Opus に任せた場合でも「プランニングの要旨・判断軸をコンテキストウィンドウ頼りにする」傾向がある（人間的なミス）
- コンテキストが圧縮された瞬間、思想が失われて Opus 同士でも引き継げない状態に

**示唆**: テンプレート（plan.md 系）は **「手順の精緻さ」と「思想の保存」を両立**させる必要がある。現状のテンプレは前者寄りになっている可能性。
改善候補として、plan.md テンプレートに「判断軸・採用/不採用の理由」を明示する欄を追加することが考えられる（将来の Issue 化候補）。

##### 補足: 別 AI / 別セッションによるプランレビューの可能性

**動機（重要）**: レビューを AI 化したい最大の理由は **「人間（承認者）がボトルネックになっている」** 問題。

- 自分の確認待ち ≒ 組織の上司の承認待ち・顧客の回答待ちと同じ構造
- トヨタ生産方式・リーンソフトウェア開発では **「待ち」はムダ** と明確に位置づけられる
- せっかく AI チームとして役割分担しているのだから、**別の役割（別 AI / 別セッション）にレビューさせる**のが筋
- 全ての確認を人間がハンドルすると、チームの並列性が人間の帯域に縛られる

今回の失敗は、**プラン確定前に別 AI / 別セッション（＝コンテキストウィンドウが異なる視点）にレビューしてもらっていたら防げた可能性**がある。

期待できる質問例:

- 「これどういうこと？」（曖昧さの指摘）
- 「ここ矛盾しているよね、なぜ？」（論理整合性）
- 「なんか書いていない前提を置いている？ 実作業する AI がわからないんじゃない？」（暗黙前提の炙り出し）

**制約**:

- プランだけ渡してレビューさせても、前提資料がないと有効性が落ちる
- Opus に「前提資料として何を一緒に渡すべきか」まで含めて整理させておけば、レビュー AI が機能する
- プロンプトの質が最終的な効きを決める

これは **Issue 化候補**：既存テンプレのどこに「プラン別視点レビュー」を組み込むか検討する価値あり。
関連する既存 Issue: [#15](https://github.com/hirobirofran/ai-team-workflow/issues/15) (Skill 化), [#22](https://github.com/hirobirofran/ai-team-workflow/issues/22) (前提を疑うテンプレ)。

##### 補足: 素振り・試運転・シミュレーション（レビューとは別軸）

プラン確定前の検証手段として、**レビューとは別軸**で「素振り／試運転／シミュレーション」も有効。

**レビューとの違い**:

| 軸 | 問いかけ | 視点 |
|---|---------|------|
| レビュー | 「このプランに抜け・矛盾はないか？」 | **外部視点**（プランの健全性を外から評価） |
| 素振り・試運転 | 「これをきみにお願いしたとして、どうやる？できそう？わからないところは？」 | **実行者視点**（プランの実行可能性を内側から評価） |

どちらも「実行前に問題を検出する」機能は同じだが、角度が違う。仕組みはほぼ同じ（別 AI / 別セッションに投げる）だが、**プロンプトと返答の質が別物**になる。

**返答の質が鍵**:

- ❌ 望ましくない返答: 「できます！」「なんちゃらかんちゃら」と調子よく答える（Gemini にありがち）
- ✅ 望ましい返答: 「できないです。自分が触るファイルにアクセスできないので」「ここの前提が不明です」といった**実行前の具体的障害や不明点の明示**
- プロンプトで具体性を引き出す必要あり（「できる／できない」ではなく「どこで詰まりそうか・何があれば進めるか」を聞く）

**実装候補フロー**:

- 案 A: **GitHub Copilot Plan モード**で呼び出して揉む → そのプランを Opus に見せてフィードバック → 「これなら回るかも」
- 案 B: 実行役想定の Sonnet/Haiku にシミュレートさせる（ただし Haiku は純シミュレーションにはトークンが無駄になりがち）
- 案 C: 新規セッション Opus に「実行者としてこのプランを評価して」と依頼

これも **Issue 化候補**（レビューとは別 Issue として切る。仕組みは共通だが目的が違うため）。

##### 補足: 「前提を疑う」のテンプレ組込み（トヨタ生産方式の類推）

この一連の失敗は、**「止まる仕組みがないまま自動化を進めた」結果**として読める。
[Issue #22](https://github.com/hirobirofran/ai-team-workflow/issues/22) で扱っている「前提を疑う」のテンプレ組込みに、以下の類推が参考になる:

**トヨタ生産方式の自動化思想（ジドウカ）**:

自動化する大前提として **「異常があれば即座に止まる仕組み」** がある。
これは豊田佐吉の自動織機（糸が切れたら自動停止）に遡る思想。

トヨタ生産方式でも継承され:

- ベルトコンベアのいたるところに **停止紐（アンドン）** がある
- スタッフは誰でも引ける、絶対に咎められない、**むしろ褒められる**文化
- ラインが止まる＝「今まさに改善が起きている」と解釈される
- 従来の「生産ラインは止めてはならない」とは**真逆の思想**

**AI チームへの類推**:

- AI が「突っ切らずに止まる・確認する・疑う」ことは**咎められるべき遅延ではなく、改善の機会**
- 「前提を疑う」「詰まったら報告」は生産性を落とす制約ではなく、**自動化を成立させる前提条件**
- この思想転換が入らないと、テンプレを増やしても「止まらずに突っ切る」行動は変わらない

この枠組みは、既存 Issue #22 の実装設計や #15 の Skill 化（ `/stop_and_report` ）の思想的背景として組み込むと、単なる指示文ではなく原則として効きやすい。

> **注記**: 上記は現時点では **仮説どまり** であり、検証が必要。トヨタ生産方式の類推がどこまで AI の挙動設計に当てはまるか、実運用ログで確認していく段階。

#### ログ: 2026-04-24 自然発生した実例（大量 PDF 情報抽出）

**文脈**: 顧客支給の大量 PDF（数十ファイル）から特定プロジェクト関連情報を抽出。顧客環境 / 社内環境の LLM が分離しており、データ持ち出し制約あり。

**各 AI の挙動比較**:

| AI | 観察された挙動 |
|---|---|
| 顧客環境 NotebookLM | プロジェクト固有情報が不足しており、重要項目を真逆に判定 |
| 社内 Gemini | 顧客 LM 向けプロンプト作成を依頼 → 詰めが甘い・時間が溶ける |
| Claude.ai Web（Opus+アダプティブ） | Gemini のやり取りを見せて調整依頼 → 「それでいいと思いますよ」と質問せず（プロンプトで質問を促しても） |
| **Claude Code**（Opus+Effort MAX+Thinking+Superpowers） | 膨大で緻密な計画＋手順書＋複数プロンプト。フェーズ1/2の二段構え。次工程（Claude Code 自身の CSV 整理）に最適化した中間出力を設計 |

**Claude Code が自発的に行った設計判断**:

- LM の出力制限を把握したプロンプトチューニング（フェーズ分割で省略を回避）
- 「フェーズ2は一度に出しますか？それともフェーズ1結果を見て微修正しますか？」と選択肢提示
- 出力形式の提案：「表形式で諸元と結果を対応付けますか？それとも箇条書き？」
  - ユーザー「いったん箇条書きで、必要になったら表に」に対して**反対意見**: 「表がお勧めです。絶対顧客から突っ込まれて諸元と照らし合わせるようになっていないと困ると思います」
- **モデル振り分けを自ら提案**:
  - LM 出力後の精査 → Sonnet (Thinking オン) で十分、トークン温存
  - CSV 加工部分 → Haiku でいける
  - 次の手の検討 → Opus に戻す

**観察**:

- モデルディスパッチャパターンが**手動運用ではなく自発提案**に進化している
- Claude Code は「専門家として反対意見を述べる」挙動が取れる（RLHF で抑制されがちだがテンプレ＋Superpowers で解放）
- LM のクセ知識は Gemini も持っていたが、**実行計画への落とし込み**が Gemini と Claude Code で別物（Gemini は気づいて終わり、Claude Code は構造化＋選択肢提示＋トークン設計）

### 工数見積もりの価値

自分の工数見積もりが外れやすいので、AI に釘を刺してほしい。
全フェーズでの見積もりと、実装中の残工数更新があると助かる。

### AI チームワークフローと組織設計の同型性（観察 2026-04-25）

**仮説**: AI チームワークフローの構造は人間の組織設計と同型。
どちらも **「認知リソース（ワーキングメモリ）の有限性」** という共通制約への対処として発生する。

| 対比軸 | 人間の組織 | AI チームワークフロー |
|--------|-----------|--------------------|
| ワーキングメモリ | 短期記憶 ~7 ± 2 項目 | コンテキストウィンドウ 200K〜1M tokens |
| 容量超過時 | 集中力低下・ミス増加・思考のループ | 自動コンパクト・文脈喪失・迷走 |
| 切替コスト | コンテキストスイッチ（再ウォームアップ必要） | キャッシュミス・文脈再読込 |
| 対策 | 役割分担＋外部記憶（メモ・議事録・仕様書） | セッション分離＋handoff/ ファイル |
| 調整機構 | 会議体・レビュー・1on1 | 人間によるプラン承認・PR レビュー |

**このリポジトリの設計決定との対応**:

- `handoff/issue-XX/` 構造 ≒ 案件ごとのドキュメント保管庫
- `plan.md` / `aar.md` / `checkpoint.md` ≒ 仕様書 / 振り返り / 引継ぎメモ
- 役割別テンプレート（PM / 設計 / 実装 / リサーチ） ≒ 組織の職能分担
- `consistency_check.md` ≒ 定期レビュー・監査

**属人化（単一個体依存）の AI 版**:

組織論でいう「属人化」（= あの人にしかわからない状態）は、AI チームにもそのまま現れる。

- 人間の属人化: **あの人にしかわからない**（知識・経験・判断が個人の頭の中にしか残らない）
- AI の属人化: **あのときの AI（のセッション）にしかわからない**（判断軸・経緯・思想がそのコンテキストウィンドウにしか存在しない）

さらに AI の場合、**同一セッション内でもコンテキストウィンドウ溢れ（自動コンパクト）が容易に起きる**ため、「あのときの AI」という表現でしか正確に捉えられない（数時間前の同じ AI が、既に別個体のように振る舞うことがある）。

どちらも **外部化（ドキュメント化・handoff 化）** によってのみ解消可能。
このリポジトリの handoff/ と plan/aar/checkpoint テンプレは、人間組織で開発された属人化対策を AI に転用している。

**Conway の法則との関係**:

システムは組織構造を反映する（Conway）と言われるが、**組織もまた認知制約を反映**している。
つまり順序関係ではなく「認知制約が根、そこから組織とアーキテクチャが同時に生える」構造。

**理論的背景（用語の整理）**:

この「認知リソースの有限性」は経済学・組織論で **限定合理性（bounded rationality）** と呼ばれる。
広木大地『[エンジニアリング組織論への招待 〜不確実性に向き合う思考と組織のリファクタリング](https://gihyo.jp/book/2018/978-4-7741-9605-3)』では、この限界を **2つの本質的な不確実性** として整理している:

- **他人はわからない**（他者の意図・知識・状態は完全には知り得ない）
- **未来はわからない**（結果・将来の状態は完全には予測できない）

AI チームワークフローに当てはめると綺麗に対応する:

- 「他人はわからない」 ≒ 別セッションの AI のコンテキストは渡さない限り引き継がれない → **handoff/ ファイルが必要**
- 「未来はわからない」 ≒ プラン通りに進まないことを前提にする → **AAR・checkpoint で随時調整**

つまりこのリポジトリのテンプレートは、2つの不確実性に対処する情報設計として解釈できる。

#### 補足: 経験主義との関係

同書では、この2つの不確実性に対する処方として **経験主義（「やってみないとわからない」）** を提示している（論理思考の限界への対処）。
同書の冒頭の比喩：**伏せられたトランプ3枚のうちどれがハートの A か？ 正解は「全部めくる」**（シュレーディンガーの猫的に、開けてみないと確定しない）。

この発想から各種方法論が系譜的につながる:

- **アジャイル** ← 小さく作って確かめる
- **仮説検証・リーンスタートアップ** ← やってみてから判断する
- **学習ループ（PDCA / OODA）** ← 経験から更新する

AI チームワークフローに当てはめると:

- `plan.md` → 初期仮説（不完全でも OK、やる前の想定を残す）
- `handoff/` への段階的な蓄積 → 経験からの更新記録
- `aar.md` → 学習ループの「振り返り・次に何を変えるか」

**つまりこのリポジトリの構造は、限定合理性（認知制約）への対処と、経験主義（不確実性への対処）の両方を、同じ情報設計で引き受けている** という解釈が可能。

**示唆**:

- 個人開発でも「一人で全部やる」は人間版マルチタスクで生産性が落ちる（＝ AI で Opus 一人に全部させる状態と等価）
- 逆に言えば、組織運営の知恵（議事録・仕様書・レビュー・ハンドオフ）は AI チームにそのまま転用できる
- このリポジトリが「AI チーム」と呼んでいるのは比喩ではなく、**同じ制約に対する同じ構造の解**

### テンプレが実際に効いた証拠（実地観察 2026-04-24）

AI チームテンプレートを置いているこのリポジトリで **新規セッション** を開始した際、
Claude Code が以下のような挙動を自発的に取った:

- 出力 MD のフォーマットがリポジトリ内テンプレートに自然に寄った
- 「ここが不明瞭なので顧客や PM に最優先で確認してください」と**ベテラン級の確認行動**
- 与えられた情報だけで閉じず、不足情報を指摘する「こじ開け」挙動
- モデル振り分け・トークン温存・次工程最適化を**自律的に提案**（ネゴシエーションレベル）

**推定メカニズム**:

- `CLAUDE.md` → `@ai-team/NOTE.md` 経由で以下が自動ロードされる
  - 「推測での断定禁止」「セーフティネット」
  - クリティカル操作の事前確認
  - 「同じアプローチを繰り返さない」
- Superpowers プラグイン + Thinking オンで深掘りモードが維持される
- `ai-team/templates/researcher.md` の「一次情報で裏を取る」「DeepResearch 用プロンプト生成」パターンも文脈として効く

**示唆**:

- テンプレートは単なるチェックリストではなく、**モデル挙動を変えるシステムプロンプト拡張**として機能している
- リポジトリに置くだけで、新規セッションでも自動的に「専門家モード」に入る
- これは vision.md の成功定義（「1日中 AI と格闘して消耗することがなくなる」）の実地達成証拠
- 他リポジトリへの展開でも同じ効果が期待できる（要検証）

**注意点**:

- ツール使用可否が深掘りモードのトリガーになっている仮説あり。同じ Opus でも Claude.ai Web では諦めモード（「それでいいと思いますよ」）になりやすい
- Claude Code のように「調べる・試す」ができる環境でテンプレ効果が最大化される

- レビュー指摘の **トレース可能化**（plan_review.md 末尾に `→ 反映済み（plan.md SectionX.Y）` を追記）は、後続セッションでも経緯が辿れる。Issue #25 設計のテンプレ要素にすべき

### 「前提を疑う」が AI 単体で実施しづらい（実地観察 2026-04-25 #30 起票）

[Issue #30](https://github.com/hirobirofran/ai-team-workflow/issues/30)（テンプレに RSS ウォッチ機能が欲しい）の起票過程で、Claude Code（私）の最初のドラフトに 3 つの前提崩れがあり、ユーザー指摘で検出された。

| # | 既存ナレッジで分かっていたはず | 書いてしまったこと |
|---|---|---|
| ① 配置 | 「ai-team/ はテンプレ専用、ルート汚染回避」（AAR/followup.md） | 「ai-team/ テンプレに `library/` ルート直下概念を組込む」（持ち込み先のルートを汚染する形） |
| ② 目的 | library/ = RSS 相当・pull 型・Issue 駆動ではない | 「リサーチ結果が library/ に流れる動線」（Issue 駆動の researcher.md と混ぜた） |
| ③ 様式 | （暗黙文化）Issue は問題提起だけ、具体策はリサーチで決める | 「必要なこと」チェックリストで具体策に踏み込んだ完璧 Issue |

**仮説（なぜ AI 単体で難しいか）**:

- **自動操縦**: 「Issue 起票」というタスクに集中するほど、「前提を疑え」のステップが通過されてしまう。タスク集中度が高まるほどナレッジ側の制約は弱まる
- **内蔵前提の不可視性**: ③ のような AI 自身に内蔵された前提（「Issue 本文は具体的・網羅的なほど良い」）は、自分のテンプレに照らしても見えにくい。**自分の中にある前提は、自分のテンプレで検出できない構造**
- **WHY 自問頻度の低さ / 思考の深さ不足**（ユーザー観察）: 「なぜこの形で起票するのか」「なぜこの配置か」というメタ自問が、チェックリスト型処理に埋もれて発火しない。または思考量・深さ自体が足りていない可能性

**示唆**:

- これは [Issue #25](https://github.com/hirobirofran/ai-team-workflow/issues/25)（別セッションプランレビュー）の存在意義そのもの。**独立コンテキストの別 AI ／人間レビュアーでないと拾えない領域**が確実にある
- AI 単体運用では「前提を疑え」のテンプレ化だけでは効果が限定的。**構造的に外部視点を組み込む**（人間 or 別セッション）ことが現実解
- [`handoff/followup.md`](../handoff/followup.md) 「AI-hard なレビューパターン 3 点」（主語の混同 / 他者の労力の不可視化 / 自己完結と外部介入の見落とし）に **「内蔵前提の自己検出不能」** を 4 つ目候補として追加可能

**セッション内続発例（2026-04-25 同セッション）**:

「内蔵前提の自己検出不能」は本セッションで連鎖発火した。「外部視点による前提疑い」をどう実装するかを議論する流れの中で、私自身がさらに前提崩れを起こした:

- ④ **Codex 手段の誤分類**: 「`openai/codex-plugin-cc` は別 AI ベンダー = 別ツール起動 = 運用重い」と書いた直後、ユーザーから「Claude Code プラグインなのでスキル並みに呼び出し簡単」と指摘。Codex CLI ライセンス前提だが、運用コストはサブエージェント並みに軽い
- ⑤ **Gemini CLI も同様**: 「ベンダー独立 = 重い」という暗黙前提が連鎖崩壊。Gemini CLI もコマンド経由（Bash）で呼び出せるので、手動 cross-session よりずっと軽量
- ⑥ **Gemini CLI = 完全独立コンテキスト の誤り**（[20260425_gemini_review.md](../handoff/issue-28/20260425_gemini_review.md) より）: Gemini CLI 試運転で出力に「ジドウカ（自働化）」「停止ボタン」が含まれたのを「Opus と独立に同じ思想圏に到達」と解釈したが、実は Gemini CLI が CWD のファイル（特に `knowledge.md` L361/L368）を暗黙参照していた。ユーザー観察「起動フォルダ以下のファイルを読む / 常時 10 ファイル抱え込む」が裏付け。**手段マップの「独立性」評価は CWD 参照ありで実質低下する**

「**ベンダー独立 = 運用重い**」「**コマンド経由 = 真の独立コンテキスト**」という 2 つの暗黙前提が、プラグイン化・コマンド化の進歩や CLI 設計思想の差で実際には崩壊しているのに、私の内部モデルが更新されていなかった。**前提疑いの議論をしている最中ですら、自分の前提を疑えていない**という入れ子構造の実例。

**修正版・外部視点委譲手段マップ**:

| 手段 | コンテキスト独立性 | 軽量さ | 備考 |
|---|---|---|---|
| `openai/codex-plugin-cc`（プラグイン） | ★★★ | 高 | Claude Code プラグインとして呼び出し（Codex CLI ライセンス要）。プラン後コーディング委譲にも転用可能との情報あり（未検証） |

## CLAUDE.md の設計・パフォーマンス

### 行数とパフォーマンス

非公式だが 100行前後からコンテキスト消費が増え、トークン効率が落ちると言われている。
日本語は1文字 ≈ 1〜2トークンで英語より膨張しやすいため注意。

チェック方法:

```bash
wc -l CLAUDE.md   # 行数
wc -w CLAUDE.md   # 単語数（目安）
```

現状: CLAUDE.md 65行、NOTE.md 103行。合計でも許容範囲内。

### 肥大化したときの対策（軽い順）

1. **`@別ファイル` 参照**: 詳細ルールを別 md に切り出して `See @path/to/file.md` で参照。常時ロードされる点は同じだが本体をスリムにできる。このリポジトリの NOTE.md 分離が好例。
2. **コードブロックを減らす**: コードブロックはトークンを消費しやすいため、コマンド例などは簡潔に。
3. **スキルに移動**: Claude Code のカスタムスラッシュコマンド（`.claude/commands/` 以下に md ファイルを置く）として切り出すと、必要なときだけロードできる。常時読み込みの CLAUDE.md より効率的。

### スキルの使い方

`.claude/commands/review.md` を作成すると `/review` コマンドで呼び出せる。
`~/.claude/commands/` に置くと全プロジェクト共通で使える。
CLAUDE.md 本体には「〇〇は `/review` を使う」と一行書くだけでよい。

このリポジトリの `ai-team/templates/` にあるテンプレートも、`.claude/commands/` に配置してスキル化する選択肢がある（将来の改善候補）。
