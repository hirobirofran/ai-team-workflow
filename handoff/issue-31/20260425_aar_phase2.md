## 学び

- **B3「先に `gh issue create` → 採番確認 → ファイル作成」順序が機能**: プラン推定 #34/#35/#36 と実採番が一致。仮にズレても順序が逆だと破綻するので、この運用が再現性を担保する
- **依存関係明記の効果**: [#35](https://github.com/hirobirofran/ai-team-workflow/issues/35) Body と initial_prompt 両方で「[#34](https://github.com/hirobirofran/ai-team-workflow/issues/34) 完了後の方が M2 判断容易」と明記したので、後続セッションが推奨着手順を文脈なしに把握できる
- **O1 分割余地表現の効果**: [#36](https://github.com/hirobirofran/ai-team-workflow/issues/36) の Body と initial_prompt に「着手時に分割提案して構わない」と明記。決め打ちで後続セッションを縛らない設計（B1/B2 = #32/#33 と同型）
- **GitHub タスクリスト記法**: Issue #31 本文に `- [ ] #34` 形式で記載すれば GitHub のサブイシュー機能として機能、新規ラベル運用導入より軽量（B2 部分採用の妥当性）
