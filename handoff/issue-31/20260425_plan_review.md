## レビュー結果（全文）

> **対象**: [`handoff/issue-31/20260425_plan.md`](20260425_plan.md)
> **観点**: 重さ分類 / 依存順序 / サブイシュー設計 / 検証条件 / 既存ルール整合 / 抜け漏れ
> **結果**: Must 2 件 / Better 4 件 / Optional 2 件

### Must（明確に直すべき）

#### M1. P0 を「最初」と Phase 2 で「P1 → P0」が衝突している（順序の論理破綻）

`initial_prompt.md` L48 の表は **「P0 = 最初（循環防止）」** と明記。AAR L43 由来の理由「テンプレ自身が壊れたまま他のチェックを走らせると循環する」が根拠。一方、本プランは Phase 1 で先に **P1 を本セッション** で実施し、Phase 2 で P0 をサブイシュー化＝着手は別セッション送り。実時間軸では **P1 が先に commit され、P0 着手は遅延**する。「循環防止のため P0 が最初」の論理と整合しない。

ただし P1 は **整合性チェックの入力でない**（`docs/plan.md` のリンクと `followup.md` のタイポなので、`consistency_check.md` 自身は使わない）ため、循環は生じない。つまり **「P0 は他の整合性チェック実行より先」** という条件は満たされる（P1 は整合性チェックを再実行しないため）。

**提案**: プランの「判定根拠」または「Phase 1」冒頭に「**P1 は整合性チェック未使用なので循環条件外、P0 と並行可。よって本セッションで P1 着手しても循環防止違反にならない**」と一行明示する。これを書かないと、後続セッションが「P1 → P0 の順は循環防止に反する」と誤読する。

#### M2. P0 完了後の「`consistency_check.md` 自身による再チェック」が計画にない

レビュー観点 #4 の論点。`consistency_check.md` の汚染（🔴 #3）を直すのが P0 の主目的。直した後にテンプレ自身を再走させなければ、「治ったか」の verify がない。`consistency_check.md` 修正後に **再度 `general-purpose` 等で本リポを走らせて 🔴 件数が減ることを確認**するのが循環防止の論理的帰結。これが P0 サブイシュー（採番例 #34）の終了条件に **明示されていない**（initial_prompt.md L66 にも記載なし）。

**提案**: P0 サブイシュー本文 / initial_prompt.md の終了条件に「**修正後の `consistency_check.md` で本リポを再チェック → 🔴 #1, #2, #3 が解消したことを確認**」を追加。再チェック結果は `handoff/issue-34/20260425_consistency_recheck.md` 等に保存。これをやらないと「治った気」のまま閉じるリスクがある。

---

### Better（質を上げる）

#### B1. `[P0]` / `[C1]` / `[P2]` プレフィックスは既存命名規約と微妙にずれている

レビュー観点 #3。既存の Issue 命名は **`[Epic] ...`**（#31）、**`[NOTE.md 拡張] ...`**（#32）と「**カテゴリ系単語 + 説明**」型。`[P0]` `[C1]` `[P2]` は **本エピック内部の優先度ラベル**であり、Issue 一覧から見ると意味が通じない（リポ全体に P0/C1/P2 体系が無いため、初見では文脈不明）。

**提案**: タイトルに親 Issue を明示するか（例: `[#31 P0] ai-team/templates/ 旧表記統一 ...`）、もしくはラベル機能で `priority:P0` を付与し、タイトルは `[テンプレ整合性] ai-team/templates/ 旧表記統一 + consistency_check.md 汚染修正` のようにカテゴリ型に揃える。後述 B2 のラベル方針と統合判断推奨。

#### B2. ラベル付与方針が無い（観点 #6）

GitHub Issue のラベル（`enhancement` / `documentation` / `priority:*` / `epic:31` 等）を付けるかが本プランで触れられていない。Issue #31 の親子関係を一覧から辿るには、ラベルかタスクリスト（GitHub 側のサブイシュー機能）が必須。

**提案**: Phase 2 のサブイシュー作成手順に「**親 #31 を本文冒頭で参照 + `documentation` ラベル付与 + GitHub のサブイシュー機能で #34/#35/#36 を #31 の子に紐付け**」を明示。ラベル運用方針を本リポで決めていないなら、まず `gh label list` で現状確認してから決める。

#### B3. 採番ズレ時の運用注意がプランに無い（観点 #3）

L46-57 で「採番例 #34/#35/#36」と書いてあるが、実際 `gh issue create` を撃った時に他 Issue が割り込めばズレる。プランは番号前提でファイルパス（`handoff/issue-{#34}/...`）まで設計しているので、ズレた場合に **どこを書き換えるか** の運用が抜けている。

**提案**: Phase 2 の手順に「**`gh issue create` 実行 → 採番された実番号を確認 → `handoff/issue-{実番号}/` でディレクトリ作成 → initial_prompt.md 内のリンク・タイトル参照を実番号で書き直す**」を明記。または「最初に 3 連で `gh issue create` を撃って番号を確定させてからファイル群を作成する」順序にする（番号の遡及修正コストを回避）。

#### B4. P1 の verify 手順が弱い（観点 #4）

Phase 1 の作業 3 「修正前後の差分確認」だけでは、**リンクが実際に解決するか**の確認にならない。`followup.md` の `eam-workflow` → `ai-team-workflow` 修正後、修正済みリンク 4 件が **GitHub 上で 200 を返すか**を `gh api repos/hirobirofran/ai-team-workflow/issues/20` 等で 1 件ずつ叩いて確認するのが verify。`docs/plan.md` の 3 件のリンク先も `ls handoff/misc/feedback/...` で実体確認すべき。

**提案**: Phase 1 の手順に **"4. リンク verify"** を追加: ① `gh api repos/.../issues/{20,21,28}` で 200 を確認、② `ls` でパス先実体確認。これは整合性チェック 🔴 #5, #6 の根拠（「Issue リンクが全て 404 になる」）に対する完了判定。

### Optional（思想として）

#### O1. #4 P2 を「分割せずサブ化」する判断の根拠が薄い（観点 #1）

P2 は「機械的（knowledge.md 旧表記） + 判断（issue-19/ 処遇） + PR 必須（NOTE.md 追記 ※ ただし 🔴 #8 は CLAUDE.md / NOTE.md への追記なので **NOTE.md 部分だけ PR 必須**）」が混在。プランは「混在しているので 1 サブイシューにまとめる方が筋」とするが、**逆に分割した方が** ① CLAUDE.md 追記（main 直 push 可）即時消化、② NOTE.md 追記は forcing function 待ちで保留、と分業が効く可能性がある（B1/B2 と同じ構造）。

**提案**: P2 サブイシュー側で分割判断を再考できる余地を `initial_prompt.md` に残す（「**着手時に分割が筋なら新セッションで分割提案して構わない**」と明示）。プランで先に決めきる必要はないが、「混在したまま 1 サブイシュー」を所与にしない。

#### O2. Issue #31 自身の Epic close 条件と進捗管理（観点 #6）

Phase 3 L68 で「サブ 3 件 close で Epic close」運用と書いてあるが、**Epic クローズの責任者** が誰か（オーケストレーターセッションに戻る? 最後のサブイシュー担当者?）が曖昧。サブ 3 件の close タイミングがバラつくと、Epic が放置される可能性。

**提案**: Phase 2 手順に「Issue #31 本文のチェックリストに #34/#35/#36 リンクを追加し、各サブ close 時に親のチェックボックスを更新する責務を **サブ担当セッション側に課す**」と明記。Epic クローズは「全 4 サブタスクのチェックが揃ったら、最後のセッション（または次のオーケストレーター）が `gh issue close 31`」と決める。

---

### 総評

P1 を本セッションで処理 + 残 3 件サブ化の方針は妥当。重さ分類のロジックも通る。**最大の論点は M1 の順序整合性の説明不足**（プランは正しいが、論理を書き切れていないため後続が誤読する）と **M2 の verify 欠落**（P0 完了の判定基準が緩い）。Better 群（タイトル規則・ラベル・採番ズレ・リンク verify）は運用品質の問題で、入れればプランの再現性が上がる。
