### 熟成タイミングの設計：イベント駆動 vs バッチ駆動（#21 リサーチ材料）

- **イベント駆動**（例：同種ログが N 件貯まったら抽出）: 自然な圧縮・抜け漏れリスク
- **バッチ駆動**（例：月1回/フェーズ切替時に棚卸し）: カバレッジ高・過負荷リスク
- PostgreSQL の autovacuum（自動継続）vs VACUUM FULL（手動棚卸し）の類推 → **両方要る**
- #21 自動化設計の核心論点になる

### #21 リサーチフェーズに「現オープン Issue の棚卸し」を Step 0 として含める

- 自己ナレッジループを設計する前に、**既存 Issue 群がループのどの段階で詰まっているか**を整理
- 受け皿の構造を把握してから新しいループを設計する方が筋が通る
- #22/#25/#27 の統合判断にも根拠が入る

- 「困ったときに参照される側に回る、ログ蓄積の副産物として埋まっていく」方針は library/ 限定ではない
- **網羅 push は認知制約で必ず破綻する**という一般原則
- knowledge.md 自体にも適用すべき（編集者の負担管理）

### 「人間レビュアーが拾えた / AI レビュアーでは拾いにくい」パターン（#25 境界マーカー）

今回の #20 訂正往復で Claude.ai が明確にできた、**AI-hard な 3 つのレビューパターン**:

1. **主語の混同** — 私と誰かの思考の境界を見失う
2. **他者の労力の不可視化** — テンプレ化の副産物のように描いて、実は手作業の積み重ねだった点を埋没させる
3. **自己完結と外部介入の見落とし** — 人間レビュアーが拾った指摘を「自己完結」の形容で説明してしまう

### Issue 分割の判断パターン（2026-04-25 #28 起点）

ある作業が **「上位 Issue の PoC」** と **「独立した調査・成果」** の両方の性質を持つとき：

- 独立した新 Issue を起票（メイン：独立価値、副次：上位 Issue の PoC）
- 上位 Issue にコメントで関連を明示
- `handoff/` も新 Issue 番号のフォルダで管理
- 効果：後の検索性向上、PoC ループの目的が明確、上位 Issue のスコープがクリーンに保たれる
- 実例：[#28](https://github.com/hirobirofran/eam-workflow/issues/28)（Gemini CLI DR 調査）が [#19](https://github.com/hirobirofran/ai-team-workflow/issues/19)（library/ 新設）の PoC を兼ねる構造

### ファイル参照 vs プロンプト全文貼付の使い分け（AI 別）

| AI | 推奨 | 理由 |
|----|------|------|
| Claude Code | **file-read**（パス指定だけで OK） | Read tool でローカル直接読める |

### 整理タイミング問題への答え（2026-04-25 ユーザー問い起点）

**問題**: ログ形式の蓄積は記録しやすいが参照しにくい。気づいたら溜まり続けている。

**核心**: 「記録の容易さ」と「参照の容易さ」はトレードオフ。記録は append-only で済むが、整理は能動的判断が要る。今までの運用は前者寄り（楽な方向に流れていた）。

**処方（autovacuum 相当）**: 専用の整理タイミングを新設するより、**既存の forcing function に組み込む**。

| 既存 forcing function | 組み込む整理タスク |
|---|---|
| AAR 作成時 | 「同じトピックの log が 3 件以上あるか？」を必ずチェック→ あれば原則抽出を提案 |
| knowledge.md / followup.md に追記する瞬間 | 同じトピックの既存エントリを検索→ 似たものが既にあれば append でなく consolidate |
| PR レビュー時（`ai-team/` 変更時） | レビュー観点に「整理候補が見えるか」を1行追加 |

**3 件以上ルール**: 同一トピックが3エントリ蓄積したら、原則抽出して「ログ→原則の2段階運用」を発動する閾値。

**VACUUM FULL 相当（棚卸し）は別途必要**: Issue [#21](https://github.com/hirobirofran/eam-workflow/issues/21) の本丸として設計する。autovacuum で日常の整理、VACUUM FULL で定期的な構造再編、の二段構え（PostgreSQL 比喩そのまま）。

### トークン消費最適化の論点（2026-04-25 セッション末尾の振り返り）

**観察**: 5h 枠で 47% 消費。普段より明確に重い。原因の分析と対策候補を残す。

**主因**:

1. 大型ファイル（`knowledge.md` +266 行）への複数ラウンド編集 → 毎ラウンドでファイル全体がコンテキストに乗る
2. Issue 起票本文が人間向けで長い（#19 #25 #26 #27 #28 全て丁寧フォーマット）
3. `plan` / `plan_review` / `followup.md` の連続生成（1000+ 行）
4. PR レビューサイクルが計4ラウンドで each ラウンドで関連ファイル再読み込み

**改善候補**:

#### A. 疑似ブランチ方式（編集対象だけ抜き出す）

- 大型ファイル（knowledge.md 等）の編集時、**該当節だけを別ファイル `handoff/issue-XX/edits/<filename>_excerpt.md` に抜き出して作業**
- レビューラウンド中はその excerpt だけがコンテキストに乗る
- 合意後に元ファイルにマージ（diff 確認）
- Git の staging area 概念を認知トークン消費に適用したもの
- 懸念：「抜かなかった部分の影響」はマージ前 diff レビューで対応

#### B. ファイル分割＋インデックス化

- `knowledge.md` を `knowledge/index.md` + 主題別ファイル群に分割
  - 例: `knowledge/ai-models.md`, `knowledge/workflow-patterns.md`, `knowledge/cognitive-constraints.md`
- 編集時に該当ファイルだけ load する設計
- → Issue [#21](https://github.com/hirobirofran/eam-workflow/issues/21) の本丸（VACUUM FULL の出力形）

#### C. Issue / plan / AAR テンプレートの圧縮

- 「人間可読性 vs AI 効率」の二択ではなく、**情報量を保ちつつ密度を上げる**
- 具体: 「ぱっと思いつくこと（仮説）」セクション任意化、「想定手順チェックリスト」を別テンプレ化、表/箇条書き重視
- → Issue [#20](https://github.com/hirobirofran/eam-workflow/issues/20)（issue/PR テンプレ）の領域

#### D. NotebookLM + DR を「ソース化された再利用可能な調査結果」として活用

- DR 単体結果（テキスト）より、NotebookLM にソース投入した状態（ノート + ソース URL）の方が情報格納形式が良い
- 再質問可能 / 検証可能 / 別 AI への引き渡しが容易
- 重いリサーチは Claude Code の自前検索より NotebookLM + DR で外部計算リソースに投げる方がコスト効率良い

#### E. DR / 外部計算リソースへの分業徹底

- Claude Code 自前リサーチ：トークン枠を消費
- DR / NotebookLM：外部リソース（Claude の枠を使わない）
- **大量リサーチが要るときは DR / NotebookLM 優先、整理＆ドキュメント化を Claude Code** という分業が黄金パターン
