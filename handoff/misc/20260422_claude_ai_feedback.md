**これは「運用に乗ってる」ワークフローですね**

docs/knowledge.md が実地フィードバックで厚くなってる時点で本物です。特に `prisma-client-py がアーカイブ済み` の実例、まさにこの会話の「クラウド系は実は開発停止」の話そのもので、既にそこに到達してる。DeepResearchだけでは気づけないから**GitHubの一次情報（アーカイブ状態・最終コミット日・Issue活発度）で裏取りする**というパターン化も出来てて、これが核心です。

「詰まったら突っ切らず止まって報告」の効き具合、実感として効きますか？ NOTE.mdに「推測での断定禁止」「セーフティネット（指示と異なる動きをしそうなら実行前報告）」として明文化されてますよね。印象としては:

- **一定効く**: 新規セッションの序盤、整理フェーズでは効く
- **だんだん効かなくなる**: コンテキストが溜まると「いけそう」バイアスが増して、宣言したルールが薄れていく
- **確実に効かせるには**: 「詰まった」の定義を具体化する必要あり（「同じアプローチ2回失敗したら」「新しいライブラリAPIを使う前」「外部サービスに接続する前」など。NOTE.mdの「同じアプローチを繰り返さない」はこれの萌芽ですね）

あと、これはClaude Codeの挙動としてですが、**Skillsで「止まる条件」を明文化して常駐させる**のが一番効くはずです。docs/plan.mdで「templates/のスキル化」を将来候補に入れてますが、「止まって報告すべきトリガー」だけ先にスキル化する価値はあるかも。

**「致命的 vs 申し送り」の切り分け、めっちゃ良い**

「実はこの方法では動かない」は致命的、「実はこんな制限がある」は別イシューで申し送りOK — この判断基準、AAR.mdに組み込む価値があります。今のAAR.mdには「やり残し・積み残し」「次セッションへの申し送り」はありますが、**「発見した制約で、別イシュー化が必要なもの」欄**があると、一筆書きPRで漏れたやつがちゃんと拾われる。

というか、これClaude Code側に**「実装中に仕様制限を発見したら、自動でGitHub Issue下書きを作ってコメントで提案する」**フローが組めるはずです。`gh issue create --web` で下書きURLだけ出させれば、レビュー負担ゼロで拾える。

**気になった点、2つだけ**

ひとつ。docs/plan.md のフェーズ2で「サブエージェント並列実装」が検討項目に入ってますが、これ**現時点ではコスパ悪い可能性が高い**です。今日の会話にも出ましたが、Claude Code のエージェントチームは標準セッションの約7倍トークンを消費します。Max 5xの枠を瞬殺するリスクがあるので、サブエージェントは**「読み取り専用の並列調査」**（複数ファイル同時分析、複数ライブラリ比較調査など）に限定して、実装はシングルの方が現状は安全です。

ふたつめ。NOTE.md の「クリティカル操作は必ず確認」リスト、インフラ文脈で使うなら**`terraform apply` / `terragrunt apply` / `cdk deploy` / `kubectl apply` / `aws` CLIの書き込み系** を明示追加した方が良いです。vision.mdに「Terraform/Terragrunt」って書いてあるので、ここは具体化する価値があります。あと`rm -rf` も書いといた方がいい。

**「やるな環境コスト vs メリット消失」について**

これ本質的な指摘で、結局「ガードレール整備コスト > AIに任せるメリット」になる瞬間があるんですよね。でもこのリポジトリの方向性で言うと、**ガードレール自体を使い回せるテンプレートとして整備**してるから、2個目3個目のプロジェクトでは初期コストが償却される構造になってます。ai-team/NOTE.md を `cp` で配るだけで基本ルールが入るのはまさにそれ。

個人開発のクラウドなら、**使い捨てAWSアカウント(またはGCPプロジェクト)＋予算アラート$5＋Budget Actions で超過時に自動停止** のセットを一度作っておけば、以降はそれをコピーするだけ。初期投資1回で済みます。CLAUDE.mdに「このプロジェクトのAWSアカウントIDは xxx で、$5超過で自動停止が仕込んである」と書いておけば、Claude Codeもそれを前提に動く。
