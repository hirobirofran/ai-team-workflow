## 起票候補 Issue

### Issue A: NOTE.md のセーフティネットを Skill 化して常駐化する

**目的**

「詰まったら突っ切らず止まって報告」の効きを、コンテキスト長に左右されず安定化させる。現状は NOTE.md のセーフティネット節に記載があるが、長時間セッションでは記載ルールが薄れる傾向があり、Skill として切り出した方が発火の持続性が高い。あわせて NOTE.md 本体のトークンも削減できる。

**背景**

- 現状、NOTE.md の「エラー発生時」「セーフティネット」節に関連記載あり
- Claude Code は Skill（`.claude/commands/` 以下）を必要時にロードする形で参照するため、CLAUDE.md / NOTE.md 常駐より効率的
- docs/plan.md の改善候補に「templates/ のスキル化」が将来項目として既に記載されている
- docs/knowledge.md「CLAUDE.md の設計・パフォーマンス」節でスキル化の考え方も整理済み

**詳細（提案）**

1. `.claude/commands/stop_and_report.md` を新規作成する

2. 以下のトリガー条件を明文化する（「詰まった」の定義）
   - 同じアプローチで 2 回以上失敗した
   - 外部サービス接続・認証まわりで想定外の挙動
   - 新しいライブラリ / API を使う前（アーカイブ状態・最終コミット日の裏取り）
   - `terraform plan` / `terragrunt plan` の差分に想定外のリソース削除が含まれる
   - 実装中に「仕様が曖昧」と感じた
   - Webhook / 認証 / シークレット取り扱い関連のコードを書こうとしている

3. トリガー該当時の報告フォーマットを固定する
   - 何をしようとしていたか
   - 何が起きたか（試した内容を時系列で）
   - 想定される原因（推測は推測と明示）
   - 次に取りうる選択肢（3 つ以内）
   - ユーザーへの質問

4. NOTE.md 本体は「止まる条件は `/stop_and_report` を参照」の一行に置き換えて本体をスリム化

5. 配置先（`~/.claude/commands/` = 全プロジェクト共通 / プロジェクトローカル）は運用判断。個人開発＋会社プロジェクト両用なら共通配置推奨

**受け入れ条件**

- [ ] `.claude/commands/stop_and_report.md` が存在する
- [ ] NOTE.md のセーフティネット節が Skill 参照に書き換わっている
- [ ] 他リポジトリで Skill が実際に発火することを 1 回以上確認している
- [ ] docs/knowledge.md に運用知見を追記

**工数見込み**: 約 1〜2h（Skill 初回作成含む）

### Issue B: AAR.md に「発見した仕様・制約（Issue 化候補）」欄を追加

**目的**

実装中に発見された「この方式では動かない（致命的）」と「こういう制限があった（申し送りで OK）」を AAR で構造的に分離・記録し、後者が確実に Issue 化されてフォローアップされるようにする。

**背景**

- 現状の AAR.md には「やり残し・積み残し」「次フェーズ・次回セッションへの申し送り」欄はあるが、一筆書き PR で拾った仕様制約・外部ライブラリ制限などが混在しがち
- 運用実績として「実はこの方法では動かない → 致命的、やり直し」「実はこんな制限があった → 別 Issue で申し送り OK」という判断軸が既に明確になっている（運用者の経験則）
- 後者を Issue 化して貯めていくと、プロジェクト全体の制約マップになる
- prisma-client-py アーカイブのような一次情報裏取りの成果も、この欄で構造化できる

**詳細（提案）**

`ai-team/templates/aar.md` に以下のセクションを追加する。

```markdown
## 発見した仕様・制約（Issue 化候補）

実装中に判明した外部制約・ライブラリ制限・運用上の注意点で、
本タスクのスコープ外だが記録・フォローアップが必要なもの。

| 種別 | 内容 | 緊急度 | 提案アクション |
|------|------|--------|----------------|
| ライブラリ制限 | 例: prisma-client-py は 2025-04-16 にアーカイブ済み | 高 | 別 Issue 起票 |
| API 制約 | 例: LINE Messaging API の webhook は HTTPS 必須 | 中 | README 追記 |
| 運用注意 | 例: Terraform state lock 競合時の手動解除手順 | 低 | knowledge.md 追記 |

**緊急度の基準**:
- 高: 今のコードが本番で動かない可能性がある → 即 Issue 起票
- 中: 他タスクで踏む可能性が高い → Issue 起票 or README 追記
- 低: ナレッジとして残せば十分 → knowledge.md 追記
```

**受け入れ条件**

- [ ] `ai-team/templates/aar.md` に新セクションが追加されている
- [ ] 既存 AAR（handoff/misc/20260418_aar.md 等）とフォーマットが衝突しない
- [ ] 運用で 1 回以上、この欄から Issue 起票が発生している
- [ ] docs/knowledge.md に「致命的 vs 申し送り」判断軸の記載を追加

**工数見込み**: 約 30min〜1h

## 起票はしないが試してほしい運用パターン

### パターン 1: 自動 Issue 下書き生成フロー

**現状の課題**

Claude Code が「これ申し送りでいいですか？」「起票しますけどいいですか？」「しました」と会話でラリーしており、対話コストが発生している。

**提案**

NOTE.md に以下のルールを追記する。

```markdown
## 仕様制約・別 Issue 候補の発見時

実装中に「別 Issue 化すべき」と判断した事項は、作業を止めず、
以下のコマンドで下書き URL を生成して報告する:

gh issue create --repo <owner>/<repo> --title "..." --body "..." --web

--web オプションで Issue 作成画面がブラウザで開かれるのみで、
実際には投稿されない。ユーザーは内容を確認して送信するだけでよい。
Claude Code 側はコマンド実行後、URL と一行要約を報告して作業を継続する。
```

**効果**

- 会話ラリーが 1 ターンで完結する
- 起票漏れがゼロになる（後で AAR から拾う必要がなくなる）
- ユーザーは下書きを読むだけで判断できる

**注意**

この運用は Issue B の AAR 欄と二重管理になる可能性がある。使い分けは:

- 実装中に発見 → `gh issue create --web` で即下書き
- セッション終了時にまとめて → AAR の「発見した仕様・制約」欄

### パターン 2: NOTE.md「クリティカル操作」リストの具体化

vision.md で Terraform/Terragrunt を主要対象としているので、現在の抽象的リストに具体例を追加する。

```markdown
## クリティカル操作（必ず事前確認）

- `git commit`（コミット内容を見せてから）
- `git push` / PR 作成
- ファイル・ディレクトリの削除
  - `rm -rf`（特にワイルドカード含む場合）
- 既存データの上書き
- **インフラ操作**
  - `terraform apply` / `terragrunt apply` / `terragrunt run-all apply`
  - `cdk deploy`
  - `kubectl apply` / `kubectl delete`
  - `aws` CLI の書き込み系（`s3 rm`, `iam delete-*`, `ec2 terminate-instances` 等）
  - `gcloud` / `az` CLI の書き込み系
- **データベース直接操作**
  - `DROP` / `TRUNCATE` / `DELETE without WHERE`
  - マイグレーション未検証の `migrate`
- 外部サービスへの書き込みリクエスト
```

**補足**

`rm -rf` 系はフックレベルで止めるのが理想（Claude Code の pre-bash hook で正規表現マッチング）。ただしフック設定は手間なので、まずは NOTE.md に明文化して「言えばわかる」状態にしておくだけでも実用上の効果は大きい。

## 余談（docs/knowledge.md 追記候補）

今日の会話で興味深かった観察:

> テンプレを使うと Gemini が賢くなる。Claude Code で曖昧プロンプトをブラッシュアップすると DeepResearch の質が変わる。

これは「**Gemini の弱点は文章読解力ではなく曖昧プロンプトへの補完能力**」という解釈ができそう。整理すると:

- Claude Code: 曖昧 → 具体化に強い（リポジトリ文脈を加味した補完）
- Gemini: 具体 → 整理・大量処理に強い（長文読解・ドキュメント統合）
- 直列でつなぐと、人間の曖昧プロンプトが高品質な仕様書に変換される

AI チーム化の本質的価値は「各 AI の強みを活かす」だけでなく「**各 AI の弱みを別の AI で補う**」点にあるかもしれない。docs/knowledge.md の「AI ごとの特性」節に観察として追記する価値がある。

## サブエージェント並列実装について（参考情報）

docs/plan.md のフェーズ 2 に「サブエージェントを使った並列実装」が検討項目として記載されているが、現時点では**コスパが悪い可能性が高い**ので注意喚起。

Claude Code 公式ドキュメントによると、エージェントチーム（teammates）を plan モードで動かすと、標準セッションの約 7 倍のトークンを消費する。理由は各 teammate が独立のコンテキストウィンドウを持ち、別の Claude インスタンスとして並列で動くため。

Max 5x プランの使用量枠を瞬殺するリスクがあるため、当面は:

- **並列実装** → シングルエージェントを推奨
- **並列調査・読み取り** → サブエージェント活用 OK（複数ファイル同時分析、複数ライブラリ比較調査など）

という使い分けが安全。将来キャッシュ共有やコンテキスト分割戦略がこなれてきたら再検討。
