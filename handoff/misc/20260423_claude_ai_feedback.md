## 🟡 議論候補（テンプレ拡張 / 新規パターン）

### prop-1: モデルディスパッチャパターンのテンプレ化

**文脈**: ユーザーは会社の Claude Code 運用で、以下のパターンを手動で回している:

1. Opus にタスクを相談
2. Opus がプラン提示 → 「これなら Haiku / Sonnet で回せます」と振り分けを提案
3. Sonnet / Haiku で実装
4. 詰まったら Opus が Gemini DeepResearch 用プロンプトを生成
5. DR 結果を Sonnet thinking-off で処理
6. 詰まったら Opus に戻す

これは実質的に **Opus = メタオーケストレーター、Sonnet/Haiku = ワーカー、Gemini DR = 外部調査員** という役割分担。
既存の `ai-team/NOTE.md` の「サブエージェント使用ルール」やリサーチャー役テンプレと親和性が高い。

**テンプレ化候補（軽い順）**:

- **候補 A（推奨）**: `docs/knowledge.md` に「モデル振り分けの判断基準」セクションを新設してログから始める
  - 実運用ログが貯まる前にテンプレ化すると外れる
  - パターンが見えてから B / C に昇格
- **候補 B**: `ai-team/NOTE.md` に「モデル選択」の節を追加（数行）
- **候補 C**: `ai-team/templates/dispatcher.md` を新設（重い / 時期尚早の可能性）

**既存テンプレとの関係**:

- `researcher.md` の「コーディング中に詰まったとき」節（Gemini DR プロンプト生成パターン）は **既にこのパターンの一部をテンプレ化済み**。ユーザーの運用はこれの拡張版
- サブエージェント定義（`.claude/agents/`）で外部 CLI をラップするパターンもあるが、エスカレーション時のコンテキスト引き継ぎロスがあるため **対話型（Opus が毎回ディスパッチ判定）の方が適している** という意見もある（外部 Claude とユーザーの対話で出た結論）

**推奨アプローチ**: 候補 A から着手。実運用ログが 1〜2 ヶ月分貯まってから B / C を判断
