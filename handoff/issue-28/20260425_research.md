## 実現方法の選択肢

### 選択肢 A: Gemini CLI のビルトイン `google_web_search` + `web_fetch` でリサーチを構成する

- **概要**: 現状の Gemini CLI に組込まれている 2 種の Web ツールを使い、エージェントループで DR 相当の多段リサーチを再現する。CLI セッション内で完結。
- **メリット**:
  - 追加実装ゼロ（最新 stable v0.39.1 で動く）
  - 一般的な Google Search grounding と URL 取得は Gemini API の grounding 機能として正式提供（[Google Search grounding 公式 docs](https://ai.google.dev/gemini-api/docs/google-search)）
- **デメリット・リスク**:
  - 「研究計画→検索→読込→統合→可視化」の **多段オーケストレーションは自前**。DR の「collaborative planning」「visualization」「最大 60 分のバックグラウンド実行」相当は再現不可
  - 結果品質が DR より明確に劣る（Issue #25165 の本文でも「現状の CLI は assumptions が多くトレーニング知識依存」と指摘されている）
- **参考**:
  - [gemini-cli web-tools tutorial](https://geminicli.com/docs/cli/tutorials/web-tools/)
  - [Gemini API: Google Search grounding](https://ai.google.dev/gemini-api/docs/grounding)
  - [Issue #25165 本文](https://github.com/google-gemini/gemini-cli/issues/25165)

### 選択肢 B: Gemini API の Deep Research Agent（Interactions API）を CLI 外から直接叩く

- **概要**: `https://generativelanguage.googleapis.com/v1beta/interactions` に対し `agent: deep-research-preview-04-2026` で POST。Python / Node / curl から呼べる。
- **メリット**:
  - DR 本来の機能をフルに使える: collaborative planning、visualization、MCP server 連携、background 実行（最大 60 分）、follow-up クエリ（`previous_interaction_id`）
  - マルチモーダル入力（画像・PDF・docs）対応
  - GA ではないが **公式 preview** なので一次情報で裏が取れる
- **デメリット・リスク**:
  - **Interactions API は public beta**。スキーマが破壊的変更される可能性（公式 docs に明記）
  - `background=True` 強制 = 同期 1 リクエストでは終わらない（20-60 分のポーリング設計が必要）
  - **Function Calling 非対応、Structured outputs 非対応**（MCP のみ）
  - CLI セッションとは独立した別プロセス。Gemini CLI のコンテキスト（GEMINI.md やセッション履歴）は自動連携しない
- **参考**:
  - [Gemini API: Deep Research Agent](https://ai.google.dev/gemini-api/docs/deep-research)（Last updated 2026-04-21）

### 選択肢 C: Deep Research API をラップする MCP サーバを自作し、Gemini CLI から呼ぶ

- **概要**: 選択肢 B の API を MCP サーバとしてラップし、Gemini CLI の MCP 拡張として登録。CLI 内から `mcp__deep_research__research(...)` のようなツール呼び出しで DR を起動。
- **メリット**:
  - **Gemini CLI のセッション内で DR が完結** する（CLI コンテキストに結果を取り込みやすい）
  - 認証管理は MCP サーバ側に集約できる
  - CLI 自体に DR が入るのを待たず、セッション統合の体験を先に得られる
- **デメリット・リスク**:
  - **MCP サーバ実装が自前**。バックグラウンド 20-60 分の長時間実行を MCP プロトコルに乗せる工夫が必要（poll/stream 設計）
  - Interactions API の preview スキーマ変更に追従するメンテ責任を負う
  - 動作する実装例が公式リポジトリには未確認（[gh search "research" 結果](https://github.com/google-gemini/gemini-cli/issues?q=research)からも DR-MCP の公式・サードパーティ実装は未発見）
- **参考**:
  - Gemini CLI README の「Extensible: MCP (Model Context Protocol) support for custom integrations」（[README](https://github.com/google-gemini/gemini-cli/blob/main/README.md)）
  - [Deep Research Agent docs - MCP server section](https://ai.google.dev/gemini-api/docs/deep-research)

## つまずきやすいポイント

1. **「Gemini CLI に DR がある」と誤解しやすい**:
   - 一般ブログ記事や DR 出力に「Gemini CLI で DR が使える」と書かれているケースに注意
   - Gemini CLI の README には DeepResearch の文字すら無い（最新時点）
   - 「Google Search grounding」は DR と別物。grounding は単発検索ベース、DR は計画→多段検索→統合→可視化の長時間エージェント
2. **API の Deep Research と Gemini Apps（Web UI）の DeepResearch の使い分け**:
   - Gemini Apps（消費者向け Web UI）の DeepResearch: 個人作業向け、UI で完結
   - Gemini API Deep Research Agent: 開発者向け、preview、Interactions API 専用、`generate_content` では呼べない
   - **同じブランドでも別 API**。両者を混同しないこと
3. **Interactions API は public beta**:
   - スキーマ破壊変更の可能性。本番運用で依存するならバージョン pin・契約確認が必要
4. **`background=True` 必須 = ポーリング必須**:
   - DR Agent は 20〜60 分かかる。同期で待つ設計はできない
   - `store=True` も同時必須（ステート保持のため）
5. **Function Calling 不可、MCP のみ**:
   - 既存の Function Calling ベースのオーケストレーションは流用不可
   - MCP サーバ実装が必要なケースが多い
6. **「Deep Research」と「Deep Think」を区別**:
   - [Issue #10462 "Deep Think Integration for Ultra Subscribers"](https://github.com/google-gemini/gemini-cli/issues/10462)（closed, Stale）は別物（推論モード）。DR と混同しない

## 推奨アプローチ（情報として）

ai-team-workflow の用途（Claude Code がリサーチ用 DR プロンプトを生成 → 人間が DR を実行 → 結果を Claude Code に戻す）に照らした **情報整理**:

- **短期で確実に動かしたいなら**: 選択肢 D を待たず、**現行ワークフロー（Web 版 Gemini Apps の DeepResearch を人間が手動で叩く）を継続** が最もコストが低い。`ai-team/templates/researcher.md` の「ブラッシュアップ依頼プロンプト」フローが既にこれに最適化されている
- **CLI 自動化に踏み込みたいなら**: 選択肢 C（MCP ラッパ自作）が最も体験価値が高いが、Interactions API preview のメンテ責任を負う覚悟が要る。当面は選択肢 B（外部スクリプトで Interactions API を叩く）で同等の自動化が可能
- **Issue #25165 の進捗を library/ で継続ウォッチ**: 公式統合が出れば選択肢 D に切り替えられる。`library/ai-tooling-ecosystem.md` の更新ループに乗せる

> 上記は **断定ではなく情報として** の整理（researcher.md「禁止事項」に従い、推奨実装は決定しない）。
