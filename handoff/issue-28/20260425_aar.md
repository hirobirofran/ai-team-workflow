## 改善提案

### library/ 運用 PoC としての所見（4 軸）

#### 1. カテゴリ妥当性

- **結論**: 暫定 1 ファイル `ai-tooling-ecosystem.md` で十分だった
- 各エントリの「このリポジトリへの影響」欄が「テンプレ群」をスコープ整理してくれるため、AI ツール / モデル / MCP を 1 ファイルに混在させても可読性は落ちなかった
- ただし将来 10 件超になったら `claude-code.md` `gemini-models.md` 等の分割が必要になる予感（push 的な分割は不要、エントリが溜まってから分ける）

#### 2. 記載フォーマット過不足

- 提示の 4 項目（YYYY-MM-DD トピック名 / 情報源 / 要点 / このリポジトリへの影響）で書けた
- **不足を感じた点**:
  - 「**確度・要再確認のステータス**」欄があると良い。preview API・需要 Issue は変動要素なので、エントリに「Status: monitoring（要再確認）」のようなマーカーを残せると後続セッションが読みやすい
  - 「**情報源**」が単一 URL を想定したフォーマット（`- **情報源**: [URL]`）だが、複数源が必要なケースが普通だった。本エントリでは複数行にしたが、フォーマット側で複数想定にした方が自然
- 「要点」の 1〜3 行制約は緩めに運用した（要点 3 行＋背景注記）。実用上問題なし

#### 3. pull 型運用との整合

- 今回は Issue #28 という具体タスクが起点だったので push 的圧力は発生しなかった
- 「他の AI ツール（Codex / Copilot 等）もエントリ書かないと library が完成しない」という焦りは抑えられた → **pull 型の動機設計は機能している**
- ただし pull 型の真価は「実際に困ったときに参照される」場面でしか検証できない。本作業はあくまで初版運用フェーズの観察にとどまる

#### 4. 運用ルール不備

- **PR 経由フロー**: 問題なし。`handoff/followup.md` 例外を使う場面はなかった
- **熟成タイミング**: 今回は `research.md` → `library/ai-tooling-ecosystem.md` を **即時昇格** した（PoC のため）。本来は [#21](https://github.com/hirobirofran/ai-team-workflow/issues/21)（自己ナレッジループ）の設計と連動して熟成期間を取るべき。**改善候補**: 「単発リサーチでも品質が高ければ即時昇格 OK」とするか、「最低 X 日／X 件貯まってから昇格」とするかの運用基準を #21 で言語化する
- `library/index.md` の「変更は PR 経由」節と `handoff/followup.md` の「直接 push 可」例外の対比は、今回の作業で明確になった。両者の文言は今のままで OK
