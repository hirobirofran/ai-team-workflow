## 実行プロンプト

```text
以下は ai-team-workflow リポジトリのある Issue 用の実装プラン（plan.md）です。
このプランを率直にレビューしてください。

観点:
- 暗黙の前提が崩れていないか
- 抜けている考慮事項がないか
- 矛盾や不整合がないか
- 思想と手順の整合性

回答フォーマット:
- Must: 実装上の不具合になりうる
- Better: 質を上げる
- Optional: 思想として深掘り
```

「Opus が既にレビュー済み」というメタ情報は **あえて渡さない**（バイアス回避目的）。

## Gemini レビュー出力

### Must: 実装上の不具合・リスクになりうる点

- **1. 既存 `library/` ファイルの存在確認**: 既に `library/` 内にファイルが存在しているように見えるため、Step 3, 4 の「新設」で破壊的に上書きしないよう、まず現状を `Read` してから差分または全体を書き込む手順を徹底
- **2. リサーチにおける「DeepResearch」の定義**: Gemini における "Deep Research" は API レベル / Web 版 Gemini Apps / SDK・CLI 実験的機能で名称・実態が混在。リサーチ時は **「どのレイヤーでの DeepResearch 呼び出しを指しているか」を明確に区別** して記録すべき

### Better: 質を上げるための提案

- **1. 検索クエリの拡張**: `deepresearch` 以外に `search`（Search tool 統合）/ `live`（Gemini Live）/ `reasoning` 等のキーワードも `gh search` に含めると DR 的挙動の最新動向を拾いやすい
- **2. トークン圧迫の「先読み判定」の具体化**: 1.3 の「重い操作の前段で checkpoint」を、操作単位の判定基準（例「`gh search` で 20 件以上の Issue がヒットしたとき」）に落とすと事故予防になる
- **3. PoC 評価軸への「更新コスト」の追加**: Step 7 の AAR 評価軸に「情報の陳腐化に対し、このフォーマットはメンテしやすいか」を加える

### Optional: 思想・戦略の深掘り

- **1. library/ における「一次情報の定義」**: README だけでなく `cmd/` 配下のフラグ定義や API 定義（proto 等）を一次情報とみなすかの判断を `research.md` に残す
- **2. 外部介入（人間レビュー）のトリガー明示**: Step 6 の self-review で根本前提が崩れていた場合、Step 8 を待たず人間に checkpoint 報告 = 「停止ボタン」意識（ジドウカ思想に近づく）

## ⚠️ 試運転後に発見された前提崩れ ⑥：Gemini CLI は CWD ファイルを暗黙参照していた

試運転完了後、ユーザー観察「Gemini CLI は起動フォルダ以下のファイルを読み、常時 10 ファイル抱え込む」を踏まえて検証したところ:

- Gemini Optional O2 で使われた「**ジドウカ（自働化）**」「**停止ボタン**」は、`docs/knowledge.md` L361「トヨタ生産方式の自動化思想（ジドウカ）」 / L368「停止紐（アンドン）」に **既出の表現**
- → 「Gemini が独立に同じ思想圏に到達」という当初解釈は **誤り**
- → 実際は Gemini が `knowledge.md` を読んで、その表現を借用した

### 修正された試運転結論

| 評価軸 | 当初判定 | 修正後 |
|---|---|---|
| 軽量さ | ✅ コマンド一発、数十秒 | ✅ 変わらず |
| 独立コンテキスト | ✅ 確認 | ⚠️ **再考要**：CWD ファイル参照あり、真の独立ではない |
| ベンダー独立性 | ★★★ | ★★★（モデル由来バイアスは独立、CWD は別軸） |
| Opus と被らない指摘の価値 | ✅ あり | ✅ あり（独自指摘 5〜6 件は事実、ただし「Gemini が CWD で読んだ範囲から Opus が拾えていない部分を抽出」か「純粋にトレーニング由来」かの切り分けは未済） |

### 真の独立性試験の手段（次回候補）

- (a) リポジトリ外（別ディレクトリ）で Gemini CLI 実行 + plan.md だけ stdin 渡し
- (b) Gemini CLI のファイル参照を切るオプション（あれば）の調査
- (c) Gemini CLI が「常時 10 ファイル抱え込む」仕様の詳細を確認（`library/ai-tooling-ecosystem.md` 更新候補）

## 試運転として判明したこと（最終）

- ✅ **コマンド一発の軽量さ**は確認できた。手動 cross-session よりはるかに低コスト
- ✅ **独自指摘の質**は Opus と同等以上、被りも少なく追加価値あり
- ✅ **事実関係系の前提崩れ検出**: DeepResearch 定義の混在を拾った（修正版マップの予測通り）
- ⚠️ **独立コンテキスト性は当初想定より低い**: CWD ファイル参照あり。真の独立性測定には別ディレクトリ実行が必要
- ⚠️ **対人文脈系（AI-hard 3 点）検出**: 直接的には拾わず（予測通り）。O2「停止ボタン」は「自己完結 vs 外部介入」の境界に肉薄したが、これも CWD ファイル由来かもしれない

### 副次発見

- Gemini CLI は `extensions` / `trustedFolders` 等の構成があり、リポジトリ統合度が高い設計
- 「常時 10 ファイル抱え込む」設計は、リポジトリ全体の文脈理解には強い反面、**「真の第三者視点」の独立コンテキスト検証には不向き**
