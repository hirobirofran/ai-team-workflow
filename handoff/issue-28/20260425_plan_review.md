## Must（実装上の不具合になりうる）

### M1. checkpoint.md 発動トリガーが抽象的

**現状**: Section 1.3 / 5 で「トークン圧迫の兆候があれば」と書いているが、判定基準がない。気づいたら手遅れになる可能性。

**反映案**:

- 具体的閾値を1つ決める。例: **コンテキストウィンドウ残 40% を下回った時点で checkpoint 化**
- リサーチで Issues を多数読み込む等、**先読みできる重い操作の前に前段で checkpoint** も併記

### M2. 「リサーチが浅い」の判定基準が曖昧

**現状**: Section 2 Step 2 末尾の「Web/gh だけでは結論が浅いと判断したら DR 用プロンプト生成」だが、誰がどう判定するか不明。

**反映案**: 以下の判定基準を明文化

- 一次情報内で結論が直接得られない（README/CHANGELOG/Issues 全部見ても明確化されない）
- 一次情報間に矛盾がある
- 最新版の状況が確認できない

このいずれか1つでも当てはまれば「浅い」と判定、DR プロンプト案を生成する。

### M3. 既存構造図を読むステップが Step 5 にない

**現状**: Section 5「README.md / CLAUDE.md の構造図に library/ を追記」だが、既存構造図のフォーマット・インデント・注釈スタイルと整合させる必要がある。先に既存を読まないと出力が浮く。

**反映案**: Step 5 の冒頭に **「Step 5.0: 既存構造図を Read で確認、書き方を踏襲」** を追加。

## Better（質を上げる）

### B1. ai-tooling-ecosystem.md を最初のカテゴリにする判断が思想に欠けている

**現状**: Section 1 の判断軸表に「なぜ index.md + ai-tooling-ecosystem.md だけで PoC するか」（他カテゴリ `claude-code.md` `models.md` 等を作らない理由）が欠落。

**反映案**: 「PoC は `library/index.md` の運用ルールを試すのが本質。最小カテゴリ1個＋実エントリ1個で十分」を 1.3 の表に追加。

### B2. AAR の「PoC 所見」観点を事前に固定

**現状**: Section 7 で「不備や改善案」と書かれているが抽象的。事前にチェックリスト化しておくと AAR で抜けが出にくい。

**反映案**: 以下4軸で AAR を書く事前定義

1. カテゴリ妥当性（暫定 1ファイル `ai-tooling-ecosystem.md` で十分か）
2. 記載フォーマット過不足（4項目で書けたか / 抜けた要素は何か）
3. pull 型運用との整合（今回の作業で pull 型が機能したか / push 的になっていなかったか）
4. 運用ルール不備（PR 経由・熟成タイミング）

### B3. Verification の relative link 確認は WebFetch ではなく Read

**現状**: Section 8 の「`library/index.md` 内のリンクが解決」を `WebFetch` で検証は不適切（ローカルファイル）。

**反映案**: Read tool で参照ファイルが存在するか確認、または markdown-link-check 系のツールを使う。

## Optional（思想として深掘り）
