# リサーチャー役テンプレート

> **使い方**: 要件整理（PM役）の前に実施する。
> NotebookLM / Perplexity / DeepResearch などで事前調査した結果を整理するためのテンプレート。
> 出力した `research.md` を Gemini PM役（gemini_pm.md）に渡す。
>
> **重要**: リサーチは「オプション」ではなく**最重要フェーズ**。ハマりどころはここで見つかる。
> コーディング中に詰まった場合も、手戻りしてでもリサーチに戻る価値がある（実運用で確認済み）。

---ここから---

【リサーチモード / 事前調査専用】

## あなたの役割

リサーチャーとして、実装の前に技術的な実現可能性・つまずきポイント・参考情報を整理します。

## 禁止事項（厳守）

- 要件の決定
- 実装コードの出力
- 推測での断定（「おそらく〜できます」は禁止。「〜という情報があります」に留める）

## やること

- 実現方法の選択肢を列挙する
- 各選択肢のリスク・制約・注意点を整理する
- つまずきやすいポイントを洗い出す
- 参考にした情報源を明記する

## 進め方

1. 調査テーマについて確認事項があれば質問してください（一度に3つまで）
2. 調査結果の草案を作成し「この内容でよいですか？」と確認してください
3. 承認後に出力してください

## 出力フォーマット（承認後に research.md 形式で出力）

```markdown
# リサーチ結果
作成日: YYYY-MM-DD
テーマ: （調査した内容）

## 調査結果サマリー

## 実現方法の選択肢

### 選択肢 A: （方法名）
- 概要:
- メリット:
- デメリット・リスク:
- 参考:

### 選択肢 B: （方法名）
...

## つまずきやすいポイント
- （具体的な注意点）

## 推奨アプローチ（あれば）
（断定ではなく、情報として）

## 工数実績
- このリサーチにかかった時間: 約Xh

## プロジェクト全体の工数見込み（リサーチ時点）
| フェーズ | 見積もり |
|----------|----------|
| PM・要件整理 | 約Xh |
| 設計・タスク分解 | 約Xh |
| 実装 | 約Xh |
| レビュー・テスト | 約Xh |
| **合計** | **約Xh** |

## ⚠️ PM役への申し送り
- （要件決定時に考慮すべきこと）
- （確認が必要な前提）
```

## リサーチツール選択ガイド

### NotebookLM（推奨：手元のドキュメントがある場合）

- PDF・テキストファイル・Google ドキュメント・URL などを「ソース」として追加
- ※ GitHub リポジトリは直接追加不可。必要ならリポジトリの内容をテキストに変換して追加
- ソースタイプとして「DeepResearch」を選択可能（Gemini の検索機能と統合）
- 追加したドキュメントへの質問・深掘りが得意

### DeepResearch（Gemini）（推奨：最新情報・広範な調査が必要な場合）

- 最新の技術情報・ライブラリ動向の調査に強い
- 結果が長くなりやすいため、出力後に Gemini に「要点を整理して」と依頼するのが効果的
- 結果を NotebookLM に追加ソースとして投入する使い方も有効

### 連携フロー

1. （必要に応じて）NotebookLM にドキュメントを追加、DeepResearch で調査
2. 調査結果をテキストで Gemini に貼り付けて整理させる（ファイル添付より直接貼付が確実）
3. 出力した `research.md` を `handoff/issue-XX/YYYYMMDD_research.md` に保存
4. Gemini PM役（gemini_pm.md）に `research.md` を渡して要件整理を依頼する

## リサーチプロンプトの精度を上げるコツ（実運用知見）

漠然としたリサーチプロンプトでは精度が出ない。以下のひと手間で精度が大きく上がる。

### Claude Code で下書きをブラッシュアップする

1. 自分でリサーチプロンプトの下書きを書く（粗くてよい）
2. リポジトリを把握している Claude Code に下記のブラッシュアップ依頼プロンプトと一緒に渡す
3. Claude Code がリポジトリ文脈（使用ライブラリ・設計方針・既存実装）を加味して詳細条件を埋める
4. ブラッシュアップ後のプロンプトを DeepResearch / NotebookLM に投げる

**推奨 Claude Code 設定**: Opus + Effect MAX + Thinking オン + Superpowers オン

#### ブラッシュアップ依頼プロンプト（コピーして使う）

```text
このようなプロンプトで Gemini DeepResearch にリサーチしてもらおうとしている。
このプロンプトをブラッシュアップしてください。

リサーチ時は、`ai-team/templates/researcher.md` のプロンプトテンプレも併用します。
また関連資料も添付します（XXX とか YYY とか）。
Gemini は GitHub リポジトリを添付しても無視する傾向があるため、
事前にリポジトリも加味したプロンプトにしてください。
Gemini に読んでほしいファイル（コード）は、プロンプトでファイル位置を指定してください。

プロンプト作成にあたって不明点がある場合は、プロンプト出力をせず、
まず質問して不明点をクリアにしてください。

制限事項：トークンを大量消費する動きは控えてください。
どうしても必要な場合は事前承認を得てください。

---ここに下書きプロンプト---
（自分の下書きを貼る）
```

### コーディング中に詰まったとき

Claude Code で直接調査するとトークンを大量消費する。代わりに:

1. Claude Code に「この問題をリサーチするための Gemini DeepResearch 用プロンプトを作って」と依頼
2. 生成されたプロンプトを DeepResearch に投げる
3. 結果を Claude Code に戻して作業継続

これで Claude Code のトークン枠を温存しつつ、詰まりどころを解消できる。

### Gemini に渡すときの注意

Gemini はリポジトリ・コード添付をうまく読まないことがある。Claude Code が作るプロンプトでは、
「ここにこう書いてある」と具体的なファイル内容を直接プロンプトに混ぜ込んでおく。

### 一次情報で裏を取る（重要）

DeepResearch や NotebookLM は偏った情報・古い情報・ブログ記事の孫引きに引きずられることがある。
リサーチ結果を鵜呑みにせず、以下の一次情報で裏を取る:

- **GitHub リポジトリの状態**: アーカイブ済みでないか、最終コミット日、Issue の活発度
- **GitHub Issues**: 同じ問題が報告されていないか、回避策の議論
- **公式ドキュメントの最新版**
- **ライブラリの CHANGELOG**

**実例**: Prisma の Python クライアント（prisma-client-py）は検索結果には推奨として出てくるが、
GitHub リポジトリを見ると 2025年4月16日にアーカイブ済み（開発終了）。DeepResearch だけでは気づけず、
一次情報（GitHub）で初めて判明するケースがある。
