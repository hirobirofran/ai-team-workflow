# PR レビューテンプレート

> **使い方**: 実装完了後、PR 作成前または作成後にレビューを依頼する。
> 「---ここから---」以下を Claude Code に貼り付ける。
> Copilot PR レビューより「複数ファイルの整合性・抜け漏れチェック」が得意。

---ここから---

## PR レビューモード

以下の手順でこの PR をレビューしてください。**修正はまだしないでください。**

### Step 1: PR の内容を把握する

```bash
gh pr view {{PR番号}} --json title,body,files
gh pr diff {{PR番号}}
```

### Step 2: タスク仕様と照合する

- `handoff/issue-XX/YYYYMMDD_task_spec.md` があれば読む
- 各タスクの受け入れ条件をすべて確認する

### Step 3: 以下の観点でレビューする

**抜け漏れチェック**:
- タスク仕様にある変更がすべて含まれているか
- 関連ファイルの変更が漏れていないか（型定義・テスト・設定ファイルなど）
- TODO/FIXME コメントが残っていないか

**整合性チェック**:
- 複数ファイルにまたがる変更が一貫しているか
- 命名・スタイルが既存コードと揃っているか
- 削除した関数・型が他のファイルで参照されていないか

**リスクチェック**:
- セキュリティ上の問題（SQLインジェクション・XSS・認証バイパスなど）
- 本番環境への影響（破壊的変更・データ損失リスク）
- 未テストの重要パス

### Step 4: 結果を以下のフォーマットで報告する

```markdown
## PR レビュー結果

**PR**: #{{番号}} {{タイトル}}
**レビュー日時**: YYYY-MM-DD

### 🔴 要対応（マージブロック）
- （修正必須の問題）

### 🟡 推奨対応（できれば直す）
- （品質向上のための指摘）

### 🟢 確認済み
- [ ] 受け入れ条件をすべて満たしている
- [ ] 関連ファイルの変更漏れなし
- [ ] セキュリティ問題なし

### 💬 コメント
- （気になった点・質問）
```

整理が完了したら「この内容でよいですか？修正を始めてよいですか？」と確認してください。
